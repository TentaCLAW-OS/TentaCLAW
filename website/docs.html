TentaCLAW OS Documentation

Everything you need to deploy, configure, and operate your AI inference cluster.

Overview #

TentaCLAW OS is an open-source operating system for AI inference. It turns scattered GPUs — across racks, rooms, or continents — into a single self-healing inference cluster with smart routing, a real-time dashboard, and an OpenAI-compatible API. Install it on bare metal or VMs, plug in your GPUs, and start serving models in minutes. No Kubernetes. No YAML mountains. Just GPUs doing work.

System Requirements #

Gateway (Control Plane)

  • OS: Ubuntu 22.04 / 24.04 LTS, or RHEL 8 / 9 (and derivatives: Rocky, Alma, CentOS Stream)
  • Runtime: Node.js 20+ (auto-installed by the installer if missing)
  • RAM: 1 GB minimum (the gateway itself is lightweight)
  • Disk: 500 MB for the application + space for the SQLite database
  • Network: Reachable by all agent nodes on port 8080 (configurable)

Agent Nodes (GPU Workers)

  • OS: Ubuntu 22.04 / 24.04 LTS, or RHEL 8 / 9
  • GPU: NVIDIA GPU with proprietary drivers installed (nvidia-smi must work)
  • Runtime: Node.js 20+
  • Network: Outbound access to the gateway on port 8080
Tip

You can run the gateway and an agent on the same machine. The gateway does not require a GPU.

Quick Install #

The fastest way to get TentaCLAW running is the one-line installer. It handles OS detection, Node.js setup, cloning, building, and systemd service creation.

bash 
curl -fsSL https://tentaclaw.io/install | bash

The installer will:

  1. Detect your OS (Debian/Ubuntu or RHEL family)
  2. Install Node.js 20+ via NodeSource if not present
  3. Install git if not present
  4. Clone (or update) the repo to /opt/tentaclaw
  5. Run npm install from the monorepo root
  6. Build the gateway: npm run build --workspace=gateway
  7. Build the dashboard: npm run build --workspace=dashboard
  8. Create a systemd service with the correct ExecStart and WorkingDirectory
Important

The gateway entry point is dist/gateway/src/index.js (not dist/index.js). This is because tsconfig uses rootDir: ".." which shifts the output directory structure.

Windows CLI #

On Windows, install the TentaCLAW CLI with PowerShell:

powershell 
irm https://tentaclaw.io/install.ps1 | iex

This installs the tentaclaw CLI globally. Point it at your gateway:

powershell 
$env:TENTACLAW_GATEWAY = "http://your-gateway:8080"
tentaclaw status

First Boot #

After installation, the gateway starts automatically via systemd. Here is what happens on first boot:

  1. Database initialization: SQLite database is created at ./data/tentaclaw.db (or the path in TENTACLAW_DB_PATH). Migrations run automatically.
  2. Cluster secret generation: A 256-bit secret is auto-generated and stored. Agents must present this secret via the X-Cluster-Secret header to register.
  3. Dashboard served: The built dashboard is served from /dashboard/ on the gateway port.
  4. Agent discovery: Agents on the same subnet will auto-discover the gateway via UDP broadcast on port 41338. Agents on remote networks need TENTACLAW_GATEWAY_URL set explicitly.
  5. GPU detection: Each agent detects NVIDIA GPUs via nvidia-smi and reports stats (VRAM, temperature, utilization) to the gateway every 10 seconds.
bash 
# Check the gateway service status
sudo systemctl status tentaclaw

# View live logs
sudo journalctl -u tentaclaw -f

# Open the dashboard
# Navigate to http://your-server-ip:8080/dashboard/

Dashboard

Accessing the Dashboard #

The dashboard is a single-page application served by the gateway at:

url 
http://your-gateway:8080/dashboard/

The dashboard connects to the gateway API and receives real-time updates via Server-Sent Events (SSE). No additional configuration is required — it auto-detects the gateway URL from the page origin.

Login & API Keys #

On first access, the dashboard will prompt you to create an admin API key. You can also set a master key via environment variable:

bash 
# Set a master API key (bypasses DB auth checks)
export TENTACLAW_API_KEY="your-secret-key-here"

# Or disable auth entirely for development
export TENTACLAW_NO_AUTH="true"

API keys are SHA-256 hashed in the database and support three scopes:

  • read — View nodes, stats, models
  • write — Deploy models, send commands, manage flight sheets
  • admin — Full access including key management and cluster configuration

Gateway API

API Overview #

The gateway exposes a RESTful API on the configured port (default 8080). All JSON responses use wrapped objects — never bare arrays. For example, the nodes endpoint returns { "nodes": [] }, not [].

url 
http://your-gateway:8080

Authentication #

The API supports two authentication methods:

API Key (Bearer Token)

Pass your API key in the Authorization header:

bash 
curl -H "Authorization: Bearer tk_your_api_key" \
  http://your-gateway:8080/api/v1/nodes

Cluster Secret (Agent Auth)

Agents authenticate using the X-Cluster-Secret header. This secret is auto-generated on first boot and stored in the database.

bash 
curl -H "X-Cluster-Secret: your-cluster-secret" \
  http://your-gateway:8080/api/v1/nodes
Security

After 5 failed authentication attempts from the same IP, that IP is blocked for 15 minutes (configurable via AUTH_BLOCK_DURATION_MS).

Endpoints #

MethodEndpointDescriptionResponse Shape
GET /api/v1/nodes List all registered nodes with latest stats { nodes: [] }
GET /api/v1/nodes/hot Nodes with GPUs above thermal threshold { hot_nodes: [], count: N }
GET /api/v1/nodes/idle Nodes with low GPU utilization { idle_nodes: [], count: N }
GET /api/v1/nodes/:nodeId Single node details { node: {} }
GET /api/v1/capacity Cluster-wide capacity summary Uses _mb suffix fields
GET /api/v1/benchmarks Benchmark results across nodes { benchmarks: [] }
GET /api/v1/models List all deployed models { models: [] }
GET /api/v1/alerts Active alerts { alerts: [] }
GET /api/v1/flight-sheets GPU assignment profiles { flight_sheets: [] }
POST /v1/chat/completions OpenAI-compatible inference endpoint OpenAI format + _tentaclaw metadata
GET /api/v1/events Server-Sent Events stream SSE stream
Note

The capacity endpoint uses _mb suffixes for memory fields (e.g., total_vram_mb, used_vram_mb), not _gb. This is by design for precision.

SSE Events #

Connect to GET /api/v1/events for real-time cluster updates. The event stream emits the following event types:

EventDescription
node_onlineA new node registered or came back online
node_offlineA node has not reported in within the stale timeout
stats_updateFresh GPU stats received from an agent
alertA new alert was triggered (thermal, error, etc.)
model_loadedA model was successfully loaded on a node
model_unloadedA model was removed from a node
javascript 
const events = new EventSource("/api/v1/events");

events.addEventListener("stats_update", (e) => {
  const data = JSON.parse(e.data);
  console.log("Node:", data.node_id, "GPUs:", data.gpus.length);
});

events.addEventListener("alert", (e) => {
  const alert = JSON.parse(e.data);
  console.log("Alert:", alert.severity, alert.message);
});
Note

The SSE client list is stored in-memory. Restarting the gateway disconnects all SSE clients. Clients should implement automatic reconnection.

OpenAI Compatibility #

The POST /v1/chat/completions endpoint is fully compatible with the OpenAI API format. You can use the official OpenAI SDK or any compatible client by pointing it at your gateway:

python 
from openai import OpenAI

client = OpenAI(
    base_url="http://your-gateway:8080/v1",
    api_key="your-api-key",
)

response = client.chat.completions.create(
    model="llama-3.1-8b",
    messages=[{"role": "user", "content": "Hello!"}],
)
print(response.choices[0].message.content)

TentaCLAW adds a _tentaclaw field to the response with routing metadata (which node served the request, latency breakdown, etc.). This field is ignored by standard OpenAI clients.

Code Agent #

tentaclaw code launches an interactive AI coding agent powered by your cluster's local models. No cloud API keys required — inference runs on your own GPUs.

Workspace

The code agent uses a workspace at ~/.tentaclaw/workspace/ containing:

  • SOUL.md — Agent personality and behavior instructions
  • USER.md — Your profile (name, role, preferences)
  • IDENTITY.md — Agent identity definition
  • MEMORY.md — Persistent memory (auto-updated by the agent)
  • AGENTS.md — Multi-agent configuration
  • TOOLS.md — Available tool definitions
  • BOOTSTRAP.md — First-run onboarding (auto-deleted after first session)

Sessions

Sessions are stored as JSONL event logs in ~/.tentaclaw/sessions/. Resume a previous session with:

bash 
tentaclaw code --resume <session-id>

Available Tools

The code agent has access to the following tools:

  • read_file — Read file contents
  • write_file — Write a complete file
  • edit_file — Surgical text replacement within a file
  • list_dir — List directory contents
  • search_files — Search file contents with regex
  • run_shell — Execute shell commands

Agent Configuration

Overview #

The TentaCLAW agent runs on each GPU node in your cluster. It collects GPU statistics (temperature, VRAM, utilization, power draw) and reports them to the gateway at a configurable interval (default: every 10 seconds). It also executes commands from the gateway such as loading models, running benchmarks, and providing WebSocket shell access.

Starting the Agent

bash 
# Navigate to the agent directory
cd /opt/tentaclaw/agent

# Start the agent
node dist/agent/src/index.js

Configuration #

Agents can be configured via environment variables or a config file at /etc/tentaclaw/rig.conf.

Config File Format

ini 
# /etc/tentaclaw/rig.conf
# TentaCLAW Agent Configuration

TENTACLAW_GATEWAY_URL=http://gateway-ip:8080
TENTACLAW_NODE_ID=gpu-node-01
TENTACLAW_CLUSTER_SECRET=your-cluster-secret
TENTACLAW_INTERVAL=10000
TENTACLAW_HOSTNAME=Octopod-1

Environment Variables #

VariableDefaultDescription
TENTACLAW_GATEWAY_URLAuto-discoverGateway URL. Required for remote networks; local subnets use auto-discovery.
TENTACLAW_NODE_IDAuto-generatedUnique identifier for this node.
TENTACLAW_FARM_HASH—Farm identifier for grouping nodes.
TENTACLAW_HOSTNAMEOS hostnameDisplay name for this node. Proxmox nodes with pve prefix are displayed as Octopods.
TENTACLAW_INTERVAL10000Stats reporting interval in milliseconds.
TENTACLAW_CLUSTER_SECRET—Must match the gateway's cluster secret.
TENTACLAW_MOCKfalseEnable mock GPU mode for development (no real GPU needed).

Auto-Discovery #

When TENTACLAW_GATEWAY_URL is not set, the agent attempts to find the gateway automatically:

  1. UDP broadcast on port 41338 — the gateway responds with its address
  2. Subnet scan as fallback — scans common ports on the local subnet (slower, approximately 5 seconds)
Tip

For production deployments, always set TENTACLAW_GATEWAY_URL explicitly. Auto-discovery is convenient for development but adds startup latency and can fail across VLANs.

Mock Mode

For development and testing without a real GPU, enable mock mode:

bash 
TENTACLAW_MOCK=true node dist/agent/src/index.js

This simulates 2 fake GPUs with random temperature, utilization, and VRAM values. Useful for testing the dashboard and API without hardware.

TentaCLAW OS (ISO)

Overview #

TentaCLAW OS is a custom Ubuntu 24.04-based Linux distribution with TentaCLAW pre-installed and pre-configured. Boot from USB or ISO, and your machine becomes an inference node immediately — no manual installation required.

Download #

Download the latest ISO from GitHub Releases:

url 
https://github.com/TentaCLAW-OS/TentaCLAW/releases

The ISO is approximately 1.6 GB and includes a live environment for testing before installation.

Features #

  • Auto-detect GPUs: NVIDIA drivers and CUDA runtime pre-installed. GPUs are detected and configured on boot.
  • Pre-configured systemd service: The TentaCLAW gateway and agent start automatically.
  • Dashboard accessible out of the box: Navigate to port 8080 after boot.
  • Minimal footprint: Stripped-down Ubuntu base — no desktop environment, office suite, or unnecessary packages.
  • Auto-updates: Optional automatic updates from the TentaCLAW repository.

Database #

TentaCLAW uses SQLite with WAL (Write-Ahead Logging) mode and foreign keys enabled. The database is stored at the path specified by TENTACLAW_DB_PATH.

bash 
# Default location
./data/tentaclaw.db

# Custom location
export TENTACLAW_DB_PATH="/var/lib/tentaclaw/cluster.db"

# In-memory (for testing)
export TENTACLAW_DB_PATH=":memory:"

Migrations are applied automatically on boot via runMigrations() in gateway/src/db/init.ts. You never need to run migrations manually.

Domain modules:

  • db/nodes.ts — Node registration and status
  • db/models.ts — Model deployment tracking
  • db/stats.ts — GPU statistics storage
  • db/auth.ts — API key management and auth state

Thermal Thresholds #

GPU temperature thresholds control alerting and the "hot nodes" classification:

VariableDefaultDescription
GPU_TEMP_CRITICAL_C90Critical temperature. Triggers immediate alert.
GPU_TEMP_HOT_C85Hot threshold. Node appears in /nodes/hot.
GPU_TEMP_WARM_C75Warm threshold. Advisory only.

Rate Limiting #

Rate limits are enforced per API key (or per IP for unauthenticated requests):

  • General API: TENTACLAW_RATE_LIMIT — default 60 requests per minute
  • Chat/Inference: TENTACLAW_CHAT_RATE_LIMIT — default 60 requests per minute

Rate-limited responses return HTTP 429 Too Many Requests with a Retry-After header.

Auth Settings #

Authentication behavior can be customized:

  • Disable all auth: TENTACLAW_NO_AUTH=true — useful for development and trusted networks
  • Master key: TENTACLAW_API_KEY — a single key that bypasses all DB-based auth checks
  • IP blocking: After 5 consecutive failed auth attempts, the IP is blocked for AUTH_BLOCK_DURATION_MS (default 15 minutes)
  • API key scopes: Keys are created with read, write, or admin scope
  • Key storage: API keys are SHA-256 hashed before storage — the plaintext is only shown once at creation time

Troubleshooting

Node Showing Offline #

If a node appears offline in the dashboard but the machine is running:

  1. Check the agent process: systemctl status tentaclaw-agent
  2. Check network connectivity: Can the agent reach the gateway on port 8080?
  3. Check the cluster secret: The agent's TENTACLAW_CLUSTER_SECRET must match the gateway's. A mismatch results in silent rejection.
  4. Check the stale timeout: The default is 90 seconds (NODE_STALE_TIMEOUT_SECS). If the agent's reporting interval is longer than this, the node will flap between online and offline.
bash 
# Verify agent connectivity
curl -s http://gateway-ip:8080/api/v1/nodes | jq .

# Check agent logs
journalctl -u tentaclaw-agent -f

GPU Not Detected #

If the agent starts but reports zero GPUs:

  1. Verify NVIDIA drivers: Run nvidia-smi. If it fails, install drivers first.
  2. Check permissions: The agent process needs access to /dev/nvidia* devices.
  3. Check for driver mismatch: Ensure the kernel module version matches the userspace driver version.
bash 
# Verify NVIDIA drivers are working
nvidia-smi

# Check kernel module
lsmod | grep nvidia

# Test with mock mode to confirm agent otherwise works
TENTACLAW_MOCK=true node dist/agent/src/index.js

Dashboard Not Loading #

If the dashboard URL returns a 404 or blank page:

  1. Check the build: The dashboard must be built with npm run build --workspace=dashboard. The Vite build outputs to gateway/public/.
  2. Verify files exist: Check that gateway/public/index.html exists.
  3. Check the gateway logs: Look for errors related to static file serving.
bash 
# Rebuild the dashboard
npm run build --workspace=dashboard

# Verify output exists
ls -la gateway/public/

# Restart the gateway
sudo systemctl restart tentaclaw

WebSocket Issues #

WebSocket connections (used for terminal access and real-time agent communication) can fail for several reasons:

  • Cluster secret mismatch: Agent WebSocket connections use X-Cluster-Secret. Dashboard connections use the session or API key. Verify both match.
  • Reverse proxy: If you have nginx or a load balancer in front of the gateway, ensure WebSocket upgrade headers are passed through.
  • Firewall: WebSocket connections use the same port as HTTP (8080) but require the Upgrade header to pass through.
nginx 
# Nginx config for WebSocket passthrough
location / {
    proxy_pass http://127.0.0.1:8080;
    proxy_http_version 1.1;
    proxy_set_header Upgrade $http_upgrade;
    proxy_set_header Connection "upgrade";
    proxy_set_header Host $host;
}

Memory Issues #

If the agent or gateway shows increasing memory usage:

  • WebSocket reconnect loop: A known issue in earlier versions caused WebSocket reconnections to leak memory. This has been fixed — ensure you are on the latest version.
  • HTTP socket pooling: The agent now properly manages HTTP socket pools to prevent connection accumulation.
  • SSE client list: The gateway's in-memory SSE client list grows with connected clients. This is normal, but disconnected clients should be pruned automatically.
  • Memory leak threshold: The agent monitors its own memory usage and logs a warning if memory grows beyond 200% of the initial baseline.
bash 
# Check agent memory usage
ps aux | grep tentaclaw

# Update to latest version
cd /opt/tentaclaw
git pull
npm install
npm run build --workspace=gateway
npm run build --workspace=agent
sudo systemctl restart tentaclaw
